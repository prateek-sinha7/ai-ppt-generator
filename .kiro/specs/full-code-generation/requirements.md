# Requirements Document

## Introduction

This feature adds a fourth generation mode — "Artisan" — to the AI Presentation Intelligence Platform and renames all four modes to user-friendly names. The existing three modes are renamed: "json" becomes "Express", "hybrid" becomes "Craft", and "code" becomes "Studio". All existing functionality is preserved under the new names. In Artisan mode, the LLM generates a single, complete, self-contained pptxgenjs JavaScript script that creates the entire presentation from scratch. Unlike Studio mode where the system creates the presentation object and slides then hands each slide to the LLM's per-slide `render_code`, Artisan mode gives the LLM total control: creating slides, choosing colors (theme is optional, not mandatory), setting presentation-level properties (masters, layouts, global defaults), and writing all rendering code as one unified script. This enables cross-slide awareness and maximum creative freedom, similar to how Claude generates presentations when used directly. Users can choose any of the four modes from the frontend selector before generating a presentation.

## Glossary

- **Artisan_Script**: A single JavaScript string generated by the LLM that contains a complete pptxgenjs program. The script receives a `pres` (PptxGenJS presentation) object as its entry point and is responsible for calling `pres.addSlide()`, configuring all slide content, and optionally setting presentation-level properties such as masters and layouts.
- **Artisan_Executor**: The pptx-service component that receives an Artisan_Script, validates it, and executes it in a sandboxed environment with the `pres` object injected, producing a complete PPTX buffer.
- **Artisan_Validator**: The pptx-service component that performs AST-based static analysis on an Artisan_Script before execution, enforcing blocked-pattern rules and a 500,000-character size limit.
- **Generation_Mode**: An enumeration with four values — "artisan" (complete script generation), "studio" (per-slide code generation), "craft" (JSON structure with embedded code snippets), and "express" (legacy JSON-to-builder).
- **Code_Validator**: The existing pptx-service component that validates per-slide `render_code` strings with a 50,000-character limit (used by Studio mode).
- **Code_Executor**: The existing pptx-service component that executes per-slide `render_code` in a sandboxed VM context (used by Studio mode).
- **Pipeline_Orchestrator**: The backend Python agent that coordinates the sequential multi-agent pipeline from industry classification through PPTX generation and visual QA.
- **Prompt_Engineering_Agent**: The backend Python agent that generates provider-specific LLM prompts, extended with an Artisan template.
- **Validation_Agent**: The backend Python agent that validates LLM output structure and content, extended to validate Artisan mode output.
- **Visual_QA_Agent**: The backend Python agent that renders slides to images, inspects them via LLM vision, and applies fixes — extended to work with Artisan mode.
- **Theme_Palette**: A JavaScript object containing named color constants and font names, optionally available in the Artisan sandbox but not mandatory.
- **pptx-service**: The Node.js Express service that builds PPTX files using pptxgenjs and provides build and preview endpoints.
- **Generation_Mode_Selector**: The frontend UI component that allows the user to choose between artisan, studio, craft, and express generation modes.
- **Progress_Indicator**: The frontend component that displays pipeline agent steps, their statuses, and mode-specific descriptions during presentation generation.

## Mode Name Mapping

| Old Internal Name | New Internal Name | User-Facing Label | Description |
|---|---|---|---|
| `json` | `express` | Express | Fastest generation |
| `hybrid` | `craft` | Craft | Balanced quality & speed |
| `code` | `studio` | Studio | Professional-grade slides |
| _(new)_ | `artisan` | Artisan | Bespoke AI-designed presentation |

## Requirements

### Requirement 1: Generation Mode Rename and Extension

**User Story:** As a platform developer, I want the GenerationMode enum to use the new mode names (express, craft, studio, artisan) and include the Artisan value, so that the system uses user-friendly names throughout and can route, validate, and render Artisan mode presentations through the pipeline.

#### Acceptance Criteria

1. THE Generation_Mode enum SHALL support four values: "artisan", "studio", "craft", and "express".
2. THE existing mode behaviors SHALL be preserved: "express" behaves identically to the former "json", "craft" behaves identically to the former "hybrid", and "studio" behaves identically to the former "code".
3. WHEN the active LLM provider is Claude, THE Pipeline_Orchestrator SHALL default the Generation_Mode to "artisan".
4. WHEN the active LLM provider is OpenAI, THE Pipeline_Orchestrator SHALL default the Generation_Mode to "studio".
5. WHEN the active LLM provider is Groq, THE Pipeline_Orchestrator SHALL default the Generation_Mode to "craft".
6. WHEN the active LLM provider is Local, THE Pipeline_Orchestrator SHALL default the Generation_Mode to "express".
7. WHEN a provider failover occurs, THE Pipeline_Orchestrator SHALL switch the Generation_Mode to match the new provider's default mode.

### Requirement 2: Artisan Prompt Template

**User Story:** As a platform developer, I want a prompt template that instructs the LLM to generate a complete, self-contained pptxgenjs script, so that the LLM has full creative control over the entire presentation including cross-slide layout decisions.

#### Acceptance Criteria

1. THE Prompt_Engineering_Agent SHALL provide an "artisan" prompt template that instructs the LLM to generate a single JavaScript function body that receives a `pres` (PptxGenJS presentation) object and creates the entire presentation.
2. THE "artisan" prompt template SHALL instruct the LLM to call `pres.addSlide()` to create each slide and to use pptxgenjs API calls (slide.addText, slide.addShape, slide.addChart, slide.addImage, slide.addTable, slide.addNotes) for all content.
3. THE "artisan" prompt template SHALL instruct the LLM that theme colors are available via an optional `theme` object but the LLM is free to choose its own hex color values.
4. THE "artisan" prompt template SHALL include the pptxgenjs API reference covering text formatting, rich text arrays, shapes, charts, tables, images, backgrounds, shadows, masters, layouts, and slide-level properties.
5. THE "artisan" prompt template SHALL include the common pitfalls section (no "#" in hex colors, no reusing option objects, use charSpacing not letterSpacing, use breakLine between array items, use bullet:true not unicode bullets).
6. THE "artisan" prompt template SHALL instruct the LLM to embed speaker notes via `slide.addNotes()` for each slide.
7. THE "artisan" prompt template SHALL instruct the LLM to return the complete script as a single string value inside a JSON wrapper: `{ "artisan_code": "<script>" }`.

### Requirement 3: Artisan Validation (AST-Based)

**User Story:** As a platform operator, I want Artisan scripts to be validated via AST-based static analysis before execution, so that malicious or broken code cannot compromise the pptx-service or the host system.

#### Acceptance Criteria

1. THE Artisan_Validator SHALL reject code that contains any of the following patterns: require(), import, process, child_process, fs, net, http, https, eval, Function constructor, setTimeout, setInterval, setImmediate, global, globalThis, __dirname, __filename.
2. THE Artisan_Validator SHALL reject code that exceeds 500,000 characters in length.
3. THE Artisan_Validator SHALL verify that the code contains at least one call to `pres.addSlide()`.
4. THE Artisan_Validator SHALL return a result object containing a valid flag and an array of error objects, each with type, message, line, and column fields.
5. THE existing Code_Validator for per-slide render_code (Studio mode) SHALL remain unchanged with its 50,000-character limit and `slide.*` API call requirement.

### Requirement 4: Artisan Sandbox Execution

**User Story:** As a platform operator, I want Artisan scripts to execute in a sandboxed environment with a 60-second timeout, so that the LLM-generated script runs safely and produces a complete PPTX buffer.

#### Acceptance Criteria

1. THE Artisan_Executor SHALL execute validated Artisan scripts within a Node.js VM sandbox with a timeout of 60 seconds for the entire script.
2. THE Artisan_Executor SHALL inject the following objects into the sandbox context: the pptxgenjs presentation object (`pres`), the Theme_Palette object (`theme`), the fonts object (`fonts`), the themes lookup object (`themes`), the `iconToBase64` async helper function, and a safe console object.
3. THE Artisan_Executor SHALL wrap the LLM code in an async IIFE that receives `pres`, `theme`, `fonts`, `themes`, and `iconToBase64` as parameters.
4. IF the Artisan_Executor encounters a runtime error, THEN THE Artisan_Executor SHALL return a structured error response containing the error message.
5. THE existing Code_Executor for per-slide render_code (Studio mode) SHALL remain unchanged with its 10-second per-slide timeout.

### Requirement 5: Artisan pptx-service Endpoints

**User Story:** As a platform developer, I want new pptx-service endpoints that accept an Artisan script and produce a PPTX file or preview images, so that Artisan mode presentations can be built and previewed.

#### Acceptance Criteria

1. THE pptx-service SHALL expose a POST `/build-artisan` endpoint that accepts a JSON body with fields: `artisan_code` (string), `design_spec` (object), and `theme` (string).
2. WHEN the `/build-artisan` endpoint receives a request, THE Artisan_Executor SHALL create a fresh pptxgenjs presentation instance, execute the artisan_code script in a sandboxed context with the presentation object injected, and write the resulting PPTX buffer.
3. THE `/build-artisan` endpoint SHALL return the PPTX buffer with Content-Type `application/vnd.openxmlformats-officedocument.presentationml.presentation` and a Content-Disposition header.
4. THE pptx-service SHALL expose a POST `/preview-artisan` endpoint that accepts the same body as `/build-artisan` and returns base64-encoded JPEG images of each slide using the same LibreOffice conversion pipeline as the existing `/preview` endpoint.
5. IF the Artisan_Validator rejects the script or the Artisan_Executor encounters a runtime error, THEN THE `/build-artisan` endpoint SHALL return a 422 response with error details and a `retry_with_studio` flag.
6. THE existing `/build`, `/preview`, `/build-code`, and `/preview-code` endpoints SHALL remain unchanged.

### Requirement 6: Artisan Fallback Behavior

**User Story:** As a platform operator, I want the system to handle Artisan generation failures gracefully, so that presentations are delivered even when the LLM produces an invalid or broken script.

#### Acceptance Criteria

1. WHEN the Artisan_Validator rejects the LLM-generated script, THE Pipeline_Orchestrator SHALL retry the LLM call once with the same prompt before falling back.
2. WHEN the Artisan_Executor encounters a runtime error, THE Pipeline_Orchestrator SHALL retry the LLM call once with the error message appended to the prompt before falling back.
3. WHEN both retry attempts fail, THE Pipeline_Orchestrator SHALL fall back to "studio" mode and re-run the content generation and validation pipeline stages using the existing per-slide code generation path.
4. THE Pipeline_Orchestrator SHALL track Artisan generation failure counts per provider using the existing CodeFailureTracker, and WHEN the failure rate exceeds 30% over the last 10 requests, THE Pipeline_Orchestrator SHALL downgrade that provider's default Generation_Mode from "artisan" to "studio".

### Requirement 7: Backend Pipeline Integration for Artisan Mode

**User Story:** As a platform developer, I want the backend pipeline orchestrator to route Artisan mode through the correct prompt template, validation, and pptx-service endpoints, so that the generation mode flows end-to-end.

#### Acceptance Criteria

1. THE Pipeline_Orchestrator SHALL pass the "artisan" Generation_Mode to the Prompt_Engineering_Agent so it selects the ARTISAN_TEMPLATE.
2. THE Pipeline_Orchestrator SHALL pass the "artisan" Generation_Mode to the Validation_Agent so it applies Artisan validation rules.
3. WHEN the Generation_Mode is "artisan", THE Pipeline_Orchestrator SHALL call the pptx-service `/build-artisan` endpoint for PPTX generation.
4. WHEN the Generation_Mode is "artisan", THE Pipeline_Orchestrator SHALL call the pptx-service `/preview-artisan` endpoint for slide image rendering.
5. THE Pipeline_Orchestrator SHALL include the "artisan" Generation_Mode in pipeline execution state so it persists across checkpoint recovery.

### Requirement 8: Validation Agent Extension for Artisan Mode

**User Story:** As a platform developer, I want the validation agent to validate Artisan mode LLM output, so that malformed scripts or missing required structure are caught before PPTX rendering.

#### Acceptance Criteria

1. WHEN the Generation_Mode is "artisan", THE Validation_Agent SHALL verify that the LLM output is a JSON object containing an `artisan_code` field with a non-empty string value.
2. WHEN the Generation_Mode is "artisan", THE Validation_Agent SHALL verify that the `artisan_code` value contains at least one call to `pres.addSlide()`.
3. THE Validation_Agent SHALL enforce a maximum `artisan_code` length of 500,000 characters.
4. THE Validation_Agent SHALL strip markdown code fences from the LLM response before parsing.
5. THE existing studio-mode and craft-mode validation logic SHALL remain unchanged.

### Requirement 9: Visual QA Agent Extension for Artisan Mode

**User Story:** As a platform developer, I want the Visual QA agent to work with Artisan generated presentations, so that the render-inspect-fix cycle catches and corrects visual issues.

#### Acceptance Criteria

1. WHEN the Generation_Mode is "artisan", THE Visual_QA_Agent SHALL use the `/preview-artisan` endpoint to render slides to images for inspection.
2. WHEN the Visual_QA_Agent identifies fixable issues on an Artisan generated presentation, THE Visual_QA_Agent SHALL instruct the LLM to produce a corrected artisan_code script, providing the original script and the issue descriptions as context.
3. THE Visual_QA_Agent SHALL pass the artisan_code script and design_spec to the `/preview-artisan` endpoint for re-rendering after fixes.
4. THE existing studio-mode and craft-mode Visual QA logic SHALL remain unchanged.

### Requirement 10: Frontend Generation Mode Selector Update

**User Story:** As a user, I want to see four clearly named generation mode options with helpful hints explaining how they differ, so that I can make an informed choice about the right quality-speed tradeoff for my presentation.

#### Acceptance Criteria

1. THE Generation_Mode_Selector SHALL display four options with the following labels and descriptions:
   - "Artisan" — "Bespoke AI-designed presentation"
   - "Studio" — "Professional-grade slides"
   - "Craft" — "Balanced quality & speed"
   - "Express" — "Fastest generation"
2. EACH mode option SHALL display a tooltip or hint on hover (desktop) or tap (mobile) that explains how that mode works and how it differs from the others. The hints SHALL be:
   - **Artisan**: "AI writes the entire presentation as one script with full creative freedom — custom colors, cross-slide consistency, and unique layouts. Best visual quality but slower."
   - **Studio**: "AI writes code for each slide individually with theme colors applied. Great visual quality with per-slide error recovery."
   - **Craft**: "Simple slides use fast templates, complex slides get AI-generated code. Good balance of speed and quality."
   - **Express**: "Uses pre-built templates for all slides. Fastest and most reliable, but limited layout variety."
3. THE Generation_Mode_Selector SHALL default to "artisan" when the active provider is Claude.
4. WHEN the user selects a mode, THE frontend SHALL include the corresponding `generation_mode` value ("artisan", "studio", "craft", or "express") in the presentation creation API request body.
5. THE user SHALL be able to select any of the four modes regardless of the active provider.

### Requirement 11: Backend API Update

**User Story:** As a platform developer, I want the backend presentation creation endpoint to accept the new mode names, so that the frontend can request any generation mode.

#### Acceptance Criteria

1. THE backend presentation creation endpoint SHALL accept "artisan", "studio", "craft", and "express" as valid values for the optional `generation_mode` field in the request body.
2. THE backend SHALL reject the old mode names ("full_code", "code", "hybrid", "json") with a validation error indicating the new names.
3. WHEN the user provides a `generation_mode` value, THE Pipeline_Orchestrator SHALL use that mode regardless of the provider default.

### Requirement 12: Adaptive Progress Indicator Update

**User Story:** As a user, I want the progress indicator to show mode-specific step descriptions with the new mode names, so that I understand what the system is doing.

#### Acceptance Criteria

1. WHILE the Generation_Mode is "artisan", THE Progress_Indicator SHALL display the Content Generation step description as "Generating complete pptxgenjs presentation script" and the Validation step as "Validating full presentation script".
2. WHILE the Generation_Mode is "studio", THE Progress_Indicator SHALL display the Content Generation step description as "Generating pptxgenjs slide code with AI" and the Validation step as "Validating generated code structure".
3. WHILE the Generation_Mode is "craft", THE Progress_Indicator SHALL display the Content Generation step description as "Generating slide content and code snippets" and the Validation step as "Validating JSON structure and code snippets".
4. WHILE the Generation_Mode is "express", THE Progress_Indicator SHALL display the existing default step descriptions.
5. THE Progress_Indicator header SHALL display the active mode name as a badge (e.g., "Artisan Mode", "Studio Mode", "Craft Mode", "Express Mode").

### Requirement 13: SSE Event Update

**User Story:** As a platform developer, I want SSE events to use the new mode names, so that the frontend can detect and display the active mode correctly.

#### Acceptance Criteria

1. THE backend SSE `agent_start` event data SHALL include the `generation_mode` field using the new names ("artisan", "studio", "craft", or "express").
2. THE frontend GenerationStatus type SHALL accept "artisan", "studio", "craft", and "express" as valid generation_mode values.

### Requirement 14: Existing Mode Preservation

**User Story:** As a platform operator, I want the existing generation mode behaviors to remain fully functional under their new names, so that no existing functionality is broken.

#### Acceptance Criteria

1. THE pptx-service SHALL continue to expose the existing `/build`, `/preview`, `/build-code`, and `/preview-code` endpoints with unchanged request and response formats.
2. THE existing Code_Validator and Code_Executor modules SHALL remain unchanged.
3. WHEN the Generation_Mode is "studio", "craft", or "express", THE Pipeline_Orchestrator SHALL use the existing prompt templates, validation rules, and pptx-service endpoints with no behavioral changes beyond the name change.
4. THE existing prompt templates (CLAUDE_TEMPLATE, CODE_TEMPLATE, HYBRID_TEMPLATE) SHALL remain unchanged in content and behavior.

### Requirement 15: Artisan Execution Performance

**User Story:** As a platform operator, I want Artisan mode PPTX generation to complete within acceptable time bounds, so that the user experience is not degraded.

#### Acceptance Criteria

1. THE Artisan_Executor SHALL enforce a 60-second timeout for the entire script execution.
2. THE `/build-artisan` endpoint SHALL complete a 15-slide presentation within 90 seconds including validation, execution, and PPTX buffer generation.
3. THE `/preview-artisan` endpoint SHALL complete a 15-slide presentation within 120 seconds including validation, execution, PPTX generation, and LibreOffice PDF/image conversion.
4. WHEN the script exceeds the 60-second execution timeout, THE Artisan_Executor SHALL terminate execution and return a timeout error.

### Requirement 16: Artisan Output Parsing

**User Story:** As a platform developer, I want robust parsing of LLM Artisan mode output, so that the system handles common LLM output formatting issues gracefully.

#### Acceptance Criteria

1. THE Validation_Agent SHALL strip markdown code fences (```json, ```javascript, ```) from the LLM response before parsing.
2. THE Validation_Agent SHALL handle the case where the LLM returns the script directly as a string (without the JSON wrapper) by wrapping it in the expected `{ "artisan_code": "<script>" }` structure.
3. THE Validation_Agent SHALL attempt JSON repair (trailing commas, missing closing brackets) with up to 2 retry attempts before declaring the output invalid.
4. FOR ALL valid Artisan mode LLM outputs, parsing then serializing then parsing SHALL produce an equivalent object (round-trip property).
