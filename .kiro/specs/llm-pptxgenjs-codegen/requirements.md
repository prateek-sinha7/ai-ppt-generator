# Requirements Document

## Introduction

This feature replaces the current JSON→builder template approach for PPTX generation with a system where the LLM generates raw pptxgenjs JavaScript code directly. The current architecture has the LLM produce structured Slide_JSON, which a ~2000-line builder.js maps to pptxgenjs API calls using ~20 hardcoded layout renderer functions. The new approach gives the LLM pixel-level control over element positioning, styling, shapes, charts, tables, and visual effects by writing actual pptxgenjs code. The system supports two modes: full code generation (LLM writes complete slide rendering code) and a hybrid mode (JSON structure with embedded pptxgenjs code snippets for complex layouts). A fallback to the existing JSON→builder pipeline is retained for less capable LLMs or when code generation fails.

## Glossary

- **Code_Executor**: The pptx-service component that receives LLM-generated pptxgenjs JavaScript code, validates it, and executes it in a sandboxed environment to produce PPTX slide content.
- **Code_Validator**: The pptx-service component that performs static analysis and runtime sandboxing on LLM-generated JavaScript code before execution, enforcing an allowlist of permitted APIs and blocking dangerous operations.
- **Slide_Code**: A string of JavaScript code generated by the LLM that makes direct pptxgenjs API calls (slide.addText, slide.addShape, slide.addChart, slide.addImage, slide.addTable) to render a single slide.
- **Slide_JSON**: The existing structured JSON format produced by the LLM, containing slide type, title, content, chart_data, table_data, comparison_data, and layout metadata.
- **Generation_Mode**: An enumeration with three values — "code" (full code generation), "hybrid" (JSON structure with embedded code snippets), and "json" (legacy JSON→builder).
- **Theme_Palette**: A JavaScript object containing named color constants (primary, secondary, accent, bg, bgDark, text, muted, chartColors, etc.) and font names injected into the LLM-generated code execution context so the LLM can reference theme colors by name.
- **Pipeline_Orchestrator**: The backend Python agent that coordinates the sequential multi-agent pipeline from industry classification through PPTX generation and visual QA.
- **Prompt_Engineering_Agent**: The backend Python agent that generates provider-specific LLM prompts, now extended with templates that instruct the LLM to produce pptxgenjs code or hybrid output.
- **Validation_Agent**: The backend Python agent that validates LLM output structure and content, now extended to validate code-mode and hybrid-mode outputs.
- **Visual_QA_Agent**: The backend Python agent that renders slides to images, inspects them via LLM vision, and applies fixes — now extended to work with code-generated slides.
- **pptx-service**: The Node.js Express service at `pptx-service/` that builds PPTX files using pptxgenjs and provides `/build` and `/preview` endpoints.
- **Design_Spec**: A data structure containing the LLM-generated or fallback design specification (colors, fonts, motif) passed through the pipeline.
- **Hybrid_Slide**: A slide object that contains standard JSON fields (type, title, speaker_notes) alongside an optional `render_code` field containing a pptxgenjs code snippet for custom rendering.
- **Generation_Mode_Selector**: The frontend UI component in the PresentationGenerator form that allows the user to choose between code, hybrid, and json generation modes before starting generation.
- **Progress_Indicator**: The frontend component that displays the pipeline agent steps, their statuses, and mode-specific descriptions during presentation generation.

## Requirements

### Requirement 1: Generation Mode Selection

**User Story:** As a platform operator, I want the system to select the appropriate generation mode (code, hybrid, or json) based on the LLM provider's capabilities, so that capable LLMs produce visually rich code-driven slides while less capable LLMs fall back to the reliable JSON→builder approach.

#### Acceptance Criteria

1. THE Pipeline_Orchestrator SHALL support three Generation_Mode values: "code", "hybrid", and "json".
2. WHEN the active LLM provider is Claude or OpenAI, THE Pipeline_Orchestrator SHALL default the Generation_Mode to "code".
3. WHEN the active LLM provider is Groq, THE Pipeline_Orchestrator SHALL default the Generation_Mode to "hybrid".
4. WHEN the active LLM provider is Local, THE Pipeline_Orchestrator SHALL default the Generation_Mode to "json".
5. WHEN a provider failover occurs from a code-capable provider to a json-only provider, THE Pipeline_Orchestrator SHALL switch the Generation_Mode to match the new provider's default mode.

### Requirement 2: Code Generation Prompt Templates

**User Story:** As a platform developer, I want LLM prompt templates that instruct the LLM to generate pptxgenjs JavaScript code for each slide, so that the LLM has pixel-level control over slide layout and styling.

#### Acceptance Criteria

1. THE Prompt_Engineering_Agent SHALL provide a "code" prompt template that instructs the LLM to generate a JavaScript function body for each slide using pptxgenjs API calls (slide.addText, slide.addShape, slide.addChart, slide.addImage, slide.addTable).
2. THE "code" prompt template SHALL include the pptxgenjs API reference covering text formatting, rich text arrays, shapes, charts, tables, images, backgrounds, shadows, and icon rendering patterns from the pptxgenjs skill documentation.
3. THE "code" prompt template SHALL include the common pitfalls section (no "#" in hex colors, no reusing option objects, use charSpacing not letterSpacing, use breakLine between array items, use bullet:true not unicode bullets).
4. THE "code" prompt template SHALL instruct the LLM to reference theme colors via a `theme` object (theme.primary, theme.secondary, theme.accent, theme.bg, theme.text, theme.chartColors) rather than hardcoding hex values.
5. THE "code" prompt template SHALL instruct the LLM to return a JSON array where each element contains slide_id, slide_number, type, title, speaker_notes, and a render_code field containing the JavaScript function body as a string.
6. THE "code" prompt template SHALL include the design rules from the existing system prompt (no accent lines under titles, dominance over equality in color weight, dark/light sandwich structure, visual motif consistency).

### Requirement 3: Hybrid Mode Prompt Templates

**User Story:** As a platform developer, I want a hybrid prompt template that combines JSON structure with optional pptxgenjs code snippets, so that moderately capable LLMs can produce enhanced layouts for complex slides while using standard JSON for simple slides.

#### Acceptance Criteria

1. THE Prompt_Engineering_Agent SHALL provide a "hybrid" prompt template that instructs the LLM to generate Slide_JSON with an optional `render_code` field on each slide.
2. WHEN the LLM includes a `render_code` field on a slide, THE pptx-service SHALL use the embedded code snippet to render that slide instead of the JSON→builder path.
3. WHEN the LLM omits the `render_code` field on a slide, THE pptx-service SHALL render that slide using the existing JSON→builder path.
4. THE "hybrid" prompt template SHALL instruct the LLM to use `render_code` for slides that benefit from custom positioning (comparison layouts, multi-chart slides, complex infographics) and omit it for standard slides (title, simple content, simple table).

### Requirement 4: Code Validation and Sandboxing

**User Story:** As a platform operator, I want LLM-generated JavaScript code to be validated and sandboxed before execution, so that malicious or broken code cannot compromise the pptx-service or the host system.

#### Acceptance Criteria

1. THE Code_Validator SHALL reject code that contains any of the following patterns: require(), import, process, child_process, fs, net, http, https, eval, Function constructor, setTimeout, setInterval, setImmediate, global, globalThis, __dirname, __filename.
2. THE Code_Validator SHALL reject code that exceeds 50,000 characters in length per slide.
3. THE Code_Validator SHALL verify that the code contains at least one pptxgenjs API call (slide.addText, slide.addShape, slide.addChart, slide.addImage, slide.addTable, or slide.background assignment).
4. THE Code_Executor SHALL execute validated code within a Node.js VM2-style sandbox with a timeout of 10 seconds per slide.
5. THE Code_Executor SHALL provide the following objects in the sandbox execution context: the pptxgenjs slide object, the pptxgenjs presentation object (for shape and chart enums), the Theme_Palette object, and the iconToBase64 helper function.
6. IF the Code_Validator rejects the code or the Code_Executor encounters a runtime error, THEN THE pptx-service SHALL return a structured error response containing the slide number, error type, and error message.

### Requirement 5: Theme Palette Injection

**User Story:** As a platform developer, I want theme color palettes injected as named constants into the code execution context, so that LLM-generated code references theme colors by name and slides remain visually consistent with the selected theme.

#### Acceptance Criteria

1. THE Code_Executor SHALL inject a `theme` object into the sandbox context containing all color values from the active Theme_Palette (primary, secondary, accent, bg, bgDark, surface, text, muted, border, highlight, chartColors).
2. THE Code_Executor SHALL inject `fonts` object into the sandbox context containing fontHeader and fontBody values from the active Design_Spec.
3. WHEN the LLM-generated code references `theme.primary`, THE Code_Executor SHALL resolve it to the 6-character hex color string from the active theme (e.g., "1A2332" for ocean-depths).
4. THE Code_Executor SHALL inject all 10 theme palettes as a `themes` lookup object so the code can reference any theme by name.

### Requirement 6: Fallback Mechanism

**User Story:** As a platform operator, I want the system to fall back to the JSON→builder approach when code generation fails, so that presentations are always delivered even when the LLM produces invalid or broken code.

#### Acceptance Criteria

1. WHEN the Code_Validator rejects the LLM-generated code for a slide, THE pptx-service SHALL attempt to parse the LLM output as Slide_JSON and render it using the existing builder.js path.
2. WHEN the Code_Executor encounters a runtime error executing a slide's code, THE pptx-service SHALL log the error and attempt to render that slide using the JSON→builder fallback if Slide_JSON fields (type, title, content) are present in the slide object.
3. WHEN all slides in a code-mode request fail validation or execution, THE pptx-service SHALL return a 422 response with per-slide error details and a flag indicating the client should retry with Generation_Mode "json".
4. THE Pipeline_Orchestrator SHALL track code generation failure counts per provider and WHEN the failure rate exceeds 30% over the last 10 requests, THE Pipeline_Orchestrator SHALL downgrade that provider's default Generation_Mode to "hybrid" or "json".

### Requirement 7: pptx-service Code Execution Endpoint

**User Story:** As a platform developer, I want a new pptx-service endpoint that accepts LLM-generated pptxgenjs code and produces PPTX files, so that code-mode and hybrid-mode slides can be rendered.

#### Acceptance Criteria

1. THE pptx-service SHALL expose a POST `/build-code` endpoint that accepts a JSON body with fields: slides (array of slide objects with render_code), design_spec (object), and theme (string).
2. WHEN the `/build-code` endpoint receives a request, THE Code_Executor SHALL create a fresh pptxgenjs presentation instance, iterate over each slide, execute the render_code in a sandboxed context, and write the resulting PPTX buffer.
3. THE pptx-service SHALL expose a POST `/preview-code` endpoint that accepts the same body as `/build-code` and returns base64-encoded JPEG images of each slide, using the same LibreOffice conversion pipeline as the existing `/preview` endpoint.
4. THE `/build-code` and `/preview-code` endpoints SHALL support hybrid-mode slides where some slides have render_code and others have standard Slide_JSON fields, routing each slide to the appropriate rendering path.
5. THE `/build-code` endpoint SHALL return the PPTX buffer with Content-Type `application/vnd.openxmlformats-officedocument.presentationml.presentation` and a Content-Disposition header.

### Requirement 8: Backend Pipeline Integration

**User Story:** As a platform developer, I want the backend pipeline orchestrator to support code generation mode throughout the pipeline, so that the generation mode flows from provider selection through prompt engineering, LLM invocation, validation, and PPTX rendering.

#### Acceptance Criteria

1. THE Pipeline_Orchestrator SHALL pass the selected Generation_Mode to the Prompt_Engineering_Agent so it selects the appropriate prompt template.
2. THE Pipeline_Orchestrator SHALL pass the Generation_Mode to the Validation_Agent so it applies the correct validation rules for the output format.
3. WHEN the Generation_Mode is "code" or "hybrid", THE Pipeline_Orchestrator SHALL call the pptx-service `/build-code` endpoint instead of `/build` for PPTX generation.
4. WHEN the Generation_Mode is "code" or "hybrid", THE Pipeline_Orchestrator SHALL call the pptx-service `/preview-code` endpoint instead of `/preview` for slide image rendering.
5. THE Pipeline_Orchestrator SHALL include the Generation_Mode in pipeline execution state so it persists across checkpoint recovery.

### Requirement 9: Validation Agent Extension

**User Story:** As a platform developer, I want the validation agent to validate code-mode and hybrid-mode LLM outputs, so that malformed code or missing required fields are caught before PPTX rendering.

#### Acceptance Criteria

1. WHEN the Generation_Mode is "code", THE Validation_Agent SHALL verify that each slide object contains slide_id, slide_number, type, title, speaker_notes, and render_code fields.
2. WHEN the Generation_Mode is "code", THE Validation_Agent SHALL verify that each render_code value is a non-empty string containing at least one pptxgenjs API call pattern (slide.addText, slide.addShape, slide.addChart, slide.addImage, slide.addTable).
3. WHEN the Generation_Mode is "hybrid", THE Validation_Agent SHALL validate slides with render_code using code-mode rules and slides without render_code using the existing Slide_JSON schema validation.
4. THE Validation_Agent SHALL enforce a maximum render_code length of 50,000 characters per slide.
5. IF the Validation_Agent detects invalid code-mode output, THEN THE Validation_Agent SHALL attempt auto-correction by stripping markdown code fences, fixing common JSON escaping issues in the render_code string, and re-validating.

### Requirement 10: Visual QA Agent Extension

**User Story:** As a platform developer, I want the Visual QA agent to work with code-generated slides, so that the render→inspect→fix cycle continues to catch and correct visual issues regardless of generation mode.

#### Acceptance Criteria

1. WHEN the Generation_Mode is "code" or "hybrid", THE Visual_QA_Agent SHALL use the `/preview-code` endpoint to render slides to images for inspection.
2. WHEN the Visual_QA_Agent identifies a fixable issue on a code-generated slide, THE Visual_QA_Agent SHALL instruct the LLM to produce a corrected render_code snippet for that slide, providing the original render_code and the issue description as context.
3. WHEN the Visual_QA_Agent identifies a fixable issue on a JSON-rendered slide in hybrid mode, THE Visual_QA_Agent SHALL apply the existing JSON-based fix logic (trim titles, reduce bullets, change layout).
4. THE Visual_QA_Agent SHALL pass the Generation_Mode and the full slide array (with render_code fields) to the preview endpoint so that re-rendering after fixes uses the correct rendering path.

### Requirement 11: User-Facing Generation Mode Selection

**User Story:** As a user, I want to see and optionally override the generation mode before starting presentation generation, so that I can choose between code mode (highest visual quality), hybrid mode (balanced), or JSON mode (fastest and most reliable) based on my needs.

#### Acceptance Criteria

1. THE frontend SHALL display a Generation Mode selector in the PresentationGenerator input form, showing three options: "Code" (labeled "Highest Visual Quality"), "Hybrid" (labeled "Balanced"), and "JSON" (labeled "Classic / Fastest").
2. THE Generation Mode selector SHALL default to the recommended mode for the active LLM provider (code for Claude/OpenAI, hybrid for Groq, json for Local).
3. WHEN the user selects a Generation_Mode, THE frontend SHALL include the selected mode in the presentation creation API request body.
4. THE backend presentation creation endpoint SHALL accept an optional `generation_mode` field in the request body and pass it to the Pipeline_Orchestrator, overriding the provider-based default.
5. IF the user does not select a mode (leaves the default), THEN THE Pipeline_Orchestrator SHALL use the provider-based default mode.

### Requirement 16: Adaptive Progress Bar

**User Story:** As a user, I want the progress bar to show mode-specific step descriptions during generation, so that I understand what the system is doing regardless of which generation mode is active.

#### Acceptance Criteria

1. WHEN the backend SSE stream emits an agent_start event, THE event data SHALL include a `generation_mode` field indicating the active mode.
2. THE frontend ProgressIndicator SHALL read the generation_mode from SSE events and adapt step descriptions accordingly.
3. WHILE the Generation_Mode is "code", THE ProgressIndicator SHALL display the Content Generation step description as "Generating pptxgenjs slide code with AI" and the Validation step description as "Validating generated code structure".
4. WHILE the Generation_Mode is "hybrid", THE ProgressIndicator SHALL display the Content Generation step description as "Generating slide content and code snippets" and the Validation step description as "Validating JSON structure and code snippets".
5. WHILE the Generation_Mode is "json", THE ProgressIndicator SHALL display the existing step descriptions with no changes.
6. THE ProgressIndicator header SHALL display the active Generation_Mode as a badge next to the title (e.g., "Generating Presentation" with a "Code Mode" badge).

### Requirement 12: Existing JSON Mode Preservation

**User Story:** As a platform operator, I want the existing JSON→builder pipeline to remain fully functional, so that it serves as the reliable fallback and continues to work for providers that do not support code generation.

#### Acceptance Criteria

1. THE pptx-service SHALL continue to expose the existing `/build` and `/preview` endpoints with unchanged request and response formats.
2. THE builder.js module SHALL remain unchanged and fully functional for JSON-mode rendering.
3. WHEN the Generation_Mode is "json", THE Pipeline_Orchestrator SHALL use the existing prompt templates, validation rules, and pptx-service endpoints with no behavioral changes.

### Requirement 13: Code Output Parsing

**User Story:** As a platform developer, I want robust parsing of LLM code-mode output, so that the system handles common LLM output formatting issues (markdown fences, extra whitespace, trailing commas) gracefully.

#### Acceptance Criteria

1. THE Validation_Agent SHALL strip markdown code fences (```json, ```javascript, ```) from the LLM response before parsing.
2. THE Validation_Agent SHALL handle render_code values that are wrapped in additional string escaping (double-escaped newlines, escaped quotes) by normalizing them before validation.
3. THE Validation_Agent SHALL attempt JSON repair (trailing commas, missing closing brackets) with up to 2 retry attempts before declaring the output invalid.
4. FOR ALL valid code-mode LLM outputs, parsing then serializing then parsing SHALL produce an equivalent object (round-trip property).

### Requirement 14: Icon Rendering in Code Mode

**User Story:** As a platform developer, I want LLM-generated code to be able to render icons from react-icons, so that code-mode slides have the same icon capabilities as the current builder.js.

#### Acceptance Criteria

1. THE Code_Executor SHALL provide an `iconToBase64` async helper function in the sandbox context that accepts an icon name string (e.g., "FaCheckCircle") and a hex color string, and returns a base64-encoded PNG data URI.
2. THE `iconToBase64` helper SHALL resolve icon names from the react-icons/fa, react-icons/md, react-icons/hi, and react-icons/bi libraries.
3. IF the `iconToBase64` helper receives an unrecognized icon name, THEN THE helper SHALL return null without throwing an error.
4. THE "code" prompt template SHALL document the iconToBase64 helper signature and available icon libraries so the LLM generates correct icon rendering calls.

### Requirement 15: Execution Performance

**User Story:** As a platform operator, I want code-mode PPTX generation to complete within acceptable time bounds, so that the user experience is not degraded compared to the current JSON→builder approach.

#### Acceptance Criteria

1. THE Code_Executor SHALL enforce a 10-second timeout per slide during sandbox execution.
2. THE `/build-code` endpoint SHALL complete a 15-slide presentation within 45 seconds including validation, execution, and PPTX buffer generation.
3. THE `/preview-code` endpoint SHALL complete a 15-slide presentation within 90 seconds including validation, execution, PPTX generation, and LibreOffice PDF/image conversion.
4. WHEN a single slide exceeds the 10-second execution timeout, THE Code_Executor SHALL terminate that slide's execution and log a timeout error for that slide.
